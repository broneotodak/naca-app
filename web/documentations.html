What is NACA, in one paragraph?

The simplest framing.

NACA = Neo's Agentic Centre Applications — Neo's personal agentic command centre. Eight specialist AI agents run 24/7 — one watches WhatsApp (Siti), one writes code (dev-agent), one reviews PRs (reviewer), one watches the fleet (supervisor), and so on. They coordinate through a shared memory database (neo-brain) and report up to Neo via WhatsApp. Neo approves big decisions ("merge this PR", "post this to LinkedIn") with a one-word reply, and the fleet handles the rest.

Built as a personal project. The end-state is a native app on Neo's phone (Flutter, currently in early build) — if it ever gets productised for a company, the whole template just gets cloned.

How NACA works · the operator stack

Four layers. Each one has a clear job. Together they form the loop that turns a single human into an effective fleet conductor.

LAYER 1 · OPERATOR

👤 Neo + Claude Code (this machine)

Neo's MacBook Pro running Claude Code CLI in the terminal. Loads a global ~/.claude/CLAUDE.md with persistent rules, plus a per-conversation auto-memory at MEMORY.md. Neo types a goal; Claude Code plans, calls tools, writes code, opens PRs.

  • Reads MEMORY.md automatically every session
  • Spawns sub-agents and remote sessions when needed
  • SSHes into Siti VPS / CLAW / NAS via Tailscale
  • Pushes to GitHub → Netlify auto-deploys
LAYER 2 · TOOLKIT

🧰 CTK · claude-tools-kit

A shared repo at ~/Projects/claude-tools-kit that holds every script, monitor, and helper used across operators and agents. Single source of truth — change a tool here and every machine picks it up.

  • tools/ — supervisor-agent, claw-command-worker, fleet-trend, save-memory, check-latest-activities
  • packages/memory/ — the @todak/memory SDK source
  • enforcement/ — CTK_ENFORCEMENT.md (rules every CC reads)
  • scripts/ — one-shots and migrations
LAYER 3 · SDK

🧠 @todak/memory · the neo-brain SDK

JS/Python client for neo-brain. Every agent — and every Claude Code session — talks to memory through this, never raw SQL. Provides search, save, credentials vault, person resolution, scheduled actions.

// JavaScript import { NeoBrain } from "@todak/memory"; const nb = new NeoBrain({ url, serviceRoleKey, agent: "my-agent" }); // Semantic search across 11K memories const hits = await nb.search("what did Lan say about Digitech?"); // Save with auto-embedding await nb.save("PR #9 merged + deployed", { category: "deploy", importance: 6, visibility: "private" }); // Vault lookup (no API key in prompts) const apiKey = await nb.getCredentialValue("netlify");
LAYER 4 · AGENTS

🤖 The specialists

Nine always-on agents you talk to most + 12 specialty/infrastructure agents that support them. All registered in agent_registry, all polling the shared agent_commands queue in neo-brain.

Plain English: there are nine agents you'd describe to a friend, and another eleven that quietly keep the machine running underneath them.

The visible nine (the ones whose names show up in your WhatsApp):

  • siti-router (Siti v2) — receives WhatsApp via siti-ingest, classifies into 1 of 8 intents, dispatches. 25 tools.
  • planner — breaks fuzzy goals into agent tasks
  • dev-agent — opens PRs (Claude Opus)
  • reviewer — verdicts on PRs (Claude Sonnet)
  • supervisor — fleet SRE on CLAW
  • timekeeper — fires scheduled jobs at fire_at
  • toolsmith — designs new tool specs
  • poster-agent — social posting (skeleton)
  • verifier-agent — confirms merged PRs actually deployed live

The supporting cast (silent infrastructure):

  • pr-decision-dispatcher — turns your "approve" WhatsApp replies into merge_pr commands. Lives on CLAW launchd, intentionally off-VPS for blast-radius isolation.
  • browser-agent · publisher-agent — social posting on Slave MBP (UI-driven via Playwright + Chromium, and API-driven via LinkedIn UGC etc.)
  • neo-twin · twin-ingest · twin-autoreply — the digital-twin pipeline that auto-replies AS Neo on his primary number (Phase 6 shadow soak)
  • naca-backend — Node.js REST API behind naca.neotodak.com, proxies to Siti's :3800
  • nas-ugreen · tr-home · backup-sync · claw-mac · neo-mbp — fleet-member heartbeat publishers (NAS, Threadripper, daily-3am rclone backup, CLAW job runner, Neo's MBP)
  • daily-checkup — fleet-health detector. Once a day at 09:00 MYT, translates the day's FAIL/WARN findings into plain English via Sonnet and sends Neo one consolidated WhatsApp digest. Phase 1 is detector-only (since 2026-05-04) — auto-fix logic is deliberately deferred until soak shows which signals are real.
The control loop. Neo says something on WhatsApp → Siti reads it → if it's a real task, Siti dispatches an agent_command → the right specialist claims it → progress goes into neo-brain → Siti notifies Neo back on WhatsApp. For risky steps (merge a PR, post publicly), Siti pauses and asks Neo to reply "approve". The whole thing runs whether Neo's at his desk or asleep.

🔌 Agent Plug & Play (engineering: registry-driven core, v2)

Refactor v1 separated code into packages (core / tools / router / interfaces). Refactor v2 separates configuration the same way: every NACA module reads its agent / project lists from agent_registry + project_registry instead of carrying hardcoded arrays. The goal is plug-and-play agent onboarding — any agent (Neo's own, or one pulled from the community / GitHub) drops into the fleet via a single registry row. Supervisor watches it, Siti labels it, naca-app accepts @-mentions, verifier checks its deploy. Zero code edits.

Plain English: today's incident (2026-05-13) — Siti went on extended leave 8 days ago, but supervisor + FCC's /api/health both kept false-alarming because each module carried its own hardcoded "what to watch" list that didn't know about the leave flag. The fix shipped same-day, but the audit revealed 4 more modules with the same disease. This section tracks the cleanup arc.

Anti-drift rule: a module MAY NOT carry an array/Set/map keyed by agent name or project repo. If such a list is needed at runtime, it MUST be derived from agent_registry or project_registry rows. The single exception is naca-mcp-bridge's exposed-tool list — that's a security boundary, not an agent list.
